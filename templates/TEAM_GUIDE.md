# 团队 AI 辅助开发指南

## 工作流程

```
需求文档 → 系统分析 → 评审确认 → 代码实现
    │           │           │
    ▼           ▼           ▼
  理解需求    架构设计     开发测试
```

## 角色职责

### 产品经理
- [ ] 在飞书写需求文档
- [ ] 明确验收标准
- [ ] 参与系统分析评审

### 开发工程师
1. [ ] 获取需求文档
2. [ ] 使用系统分析模板生成分析文档
3. [ ] 发起评审会议
4. [ ] 根据评审意见调整
5. [ ] 开始开发

### 技术负责人
- [ ] Review 系统分析文档
- [ ] 确认架构合理性
- [ ] 评估工作量
- [ ] 批准上线

## 文件位置

| 文件类型 | 位置 | 说明 |
|----------|------|------|
| 项目上下文 | `.claude/PROJECT_CONTEXT.md` | 项目基本信息 |
| 需求文档 | `.claude/requirements/` | 产品需求 |
| 系统分析 | `.claude/analysis/[需求名]-analysis.md` | 技术分析文档 |
| 项目索引 | `.claude/index/` | 代码结构索引 |
| 模板文件 | `templates/` | 各类模板 |

## 快速开始

### 首次使用（技术负责人）

```bash
# 1. 初始化项目上下文
cp templates/PROJECT_CONTEXT.md .claude/PROJECT_CONTEXT.md
# 编辑填写项目信息

# 2. 生成项目索引（可选）
# 让 AI 分析项目结构，生成索引文件

# 3. 配置飞书文档集成（如可用）
# 联系基础架构团队获取 API 权限
```

### 处理新需求（开发工程师）

```bash
# 1. 复制系统分析模板
cp templates/SYSTEM_ANALYSIS_TEMPLATE.md .claude/analysis/[需求名]-analysis.md

# 2. 使用 AI 辅助填写
# 提供需求文档，让 AI 生成初稿

# 3. 人工 Review 和调整
# 确保技术细节准确

# 4. 发起评审
```

## AI 使用技巧

### 提供上下文
```
请根据以下项目信息分析需求：
- 项目上下文：.claude/PROJECT_CONTEXT.md
- 需求文档：[粘贴或链接]
- 输出：系统分析文档初稿
```

### 验证分析
```
请检查系统分析文档：
1. 是否遗漏了受影响的模块？
2. 数据库变更是否有回滚方案？
3. 接口变更是否兼容？
4. 外部依赖是否有降级方案？
```

### 生成代码
```
根据系统分析文档，生成：
1. 数据库变更 SQL
2. 接口定义代码
3. 核心业务逻辑
4. 单元测试用例
```

## 常见问题

### Q: 如何让 AI 理解项目结构？
A: 先填写 `PROJECT_CONTEXT.md`，包含模块划分、核心接口、数据库表等信息。

### Q: 系统分析文档太长了怎么办？
A: 可以按模块拆分，每个模块一个分析文档，最后汇总。

### Q: 如何保证 AI 生成的内容准确？
A: AI 生成初稿后，必须由人工 Review 和调整，特别是技术细节。

### Q: 飞书文档无法自动获取怎么办？
A: 手动复制需求文档内容到 `.claude/requirements/` 目录。
